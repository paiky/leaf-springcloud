# 🚀 架构师进阶：微服务是如何扩容的？（单机 vs 容器 vs K8s）

在我们这套高可用抗并发系统的演进中，**“扩容” (Scaling)** 是一个极其重要的课题。当双十一大促来临，原有系统的计算能力（CPU/内存）已经跑满时，我们到底该如何迅速扩充系统规模？

在业界，扩容手段的演进经历了三个标志性的时代。为了让您有更直观的高阶架构视角，我们专门整理了这篇对比文。

---

## 时代 1：传统单机/物理机部署 (手动加人时代)

这是最初始的部署方式——把编译好的 `.jar` 文件丢到一台 Linux 服务器上，用 `nohup java -jar xxx.jar` 启动。

### 扩容是怎么做的？
1. **买机器**：向阿里云或者本地机房申请一台新的物理机/虚拟机。
2. **装环境**：运维老哥连上机器，装 Java 环境，装工具。
3. **改配置**：工程师手动修改 `application.yml` 里不兼容的配置（比如缓存池连接数）。
4. **人工部署**：把 `.jar` 传到新机器上，起进程。
5. **挂载网关**：去 Nginx (或者硬件 F5) 上手动改配置文件，把这台新机器的 IP 加入到负载分发的列表里，并执行 `nginx -s reload`。

### 痛点：
*   **慢与脆弱**：扩一台机器可能需要几小时甚至半天，而且人工配置极其容易出差错（比如端口冲突、环境少个包）。
*   **无法应对突发**：当流量像海啸一样突然涌来，等你配好机器，系统早就被冲垮了。

---

## 时代 2：纯 Docker 容器化部署 (自动化脚本时代)

当我们引入 Docker 后，服务器里的“环境”和我们的“代码”被揉在一起做成了 **镜像 (Image)**。比如我们项目里的 `Dockerfile`，就是把 JDK21 和我们的 User 服务包成了标准化集装箱。

### 扩容是怎么做的？
1. 在宿主机上，通过修改启动端口（避免相同端口冲突），再用刚才那个相同的镜像，使用不同的名字起一个新容器：
   ```bash
   docker run -d -p 8082:8080 --name leaf-service-user-2 leaf/leaf-service-user:latest
   ```
2. （如果在 Docker Compose 下）修改 `docker-compose.yml`，给微服务增加一个额外的节点宣告，再 `up -d`。

### 优势与痛点：
*   **✅ 优势**：极大地提高了发布速度！无需人工装基础软件，所有机器上的环境绝对保持 100% 一致，杜绝了“在我的电脑上没问题”的惨剧。
*   **❌ 痛点**：它依然需要你在某台具体的宿主机上去维护它！如果宿主机的硬盘撑爆了，这台机器上运行的 10 个容器还是全军覆没。你必须人工登录不同的机器集群到处敲命令，这被称为**“容器治理孤岛”**。

---

##  时代 3：Kubernetes (K8s) 云原生部署 (造物主时代)

这就是工业界的最终形态，也是我们 `infra-deploy/k8s` 目录存在的意义。K8s（Kube）不再管理“单一的容器”，它是统领千军万马（多台物理宿主机节点集群）的**云原生底层操作系统**。

### 扩容是怎么做的？(令人窒息的降维打击)

在 K8s 中，我们使用的是 **副本集缩放 (Scale)**。
不需要买机器、不需要改端口配置、不需要改网关，**只需要敲下一行指令，或者在网页上按一个按钮：**

```bash
# 将 leaf-service-user 瞬间扩容到 50 个节点应对双十一！
kubectl scale deployment leaf-service-user-deployment --replicas=50
```

K8s 收到这行命令后，**会在后台全自动执行以下魔法**：
1. **智能选址 (Scheduling)**：它会看当前集群里哪台物理宿主机（Worker Node）很闲，内存多，就把容器丢给这台机器跑。不需要人工指定跑在哪台机器上！
2. **创建与隔离 (Pod)**：利用早已在 registry 下载好的标准化镜像，像工厂印钞一样，瞬间打印出新增的 49 个隔离舱 (Pod)。
3. **服务网格动态注入 (Service/Ingress)**：不用去改 Nginx！K8s 内部的 Service 发现机制探测到新增了 49 个节点后，会自动将它们动态注册进微服务网关 (Spring Cloud Gateway / Nacos)，流量下一秒就毫无缝隙地按负载均衡砸向这 50 台机器！
4. **自愈重建 (Self-Healing)**：如果其中一台机器所在的机房网线被人拔了，K8s 会扫描并发现当前只有 49 个存活，它会立刻在另一台未断电的机器上再打印一个新的 Pod 补上，**使存活数永远等于 50！**

### 终极形态：HPA 自动弹缩
甚至连这 1 行命令都可以不用敲了！
在 K8s 里面配置一条规则（HPA）：`“如果当前 CPU 使用率超过 80%，自动加节点，最高到 100 节点；如果凌晨无人访问 CPU 掉下 20%，自动缩到 3 节点省钱。”`
系统真正具备了像呼吸一样的**弹性生命 (Elastic Compute)**！
