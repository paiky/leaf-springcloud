# 🚀 架构师进阶：微服务是如何扩容的？（单机 vs 容器 vs K8s）

在我们这套高可用抗并发系统的演进中，**“扩容” (Scaling)** 是一个极其重要的课题。当双十一大促来临，原有系统的计算能力（CPU/内存）已经跑满时，我们到底该如何迅速扩充系统规模？

在业界，扩容手段的演进经历了三个标志性的时代。为了让您有更直观的高阶架构视角，我们专门整理了这篇对比文。

---

## 时代 1：传统单机/物理机部署 (手动加人时代)

这是最初始的部署方式——把编译好的 `.jar` 文件丢到一台 Linux 服务器上，用 `nohup java -jar xxx.jar` 启动。

### 扩容是怎么做的？
1. **买机器**：向阿里云或者本地机房申请一台新的物理机/虚拟机。
2. **装环境**：运维老哥连上机器，装 Java 环境，装工具。
3. **改配置**：工程师手动修改 `application.yml` 里不兼容的配置（比如缓存池连接数）。
4. **人工部署**：把 `.jar` 传到新机器上，起进程。
5. **挂载网关**：去 Nginx (或者硬件 F5) 上手动改配置文件，把这台新机器的 IP 加入到负载分发的列表里，并执行 `nginx -s reload`。

### 痛点：
*   **慢与脆弱**：扩一台机器可能需要几小时甚至半天，而且人工配置极其容易出差错（比如端口冲突、环境少个包）。
*   **无法应对突发**：当流量像海啸一样突然涌来，等你配好机器，系统早就被冲垮了。

---

## 时代 2：纯 Docker 容器化部署 (自动化脚本时代)

当我们引入 Docker 后，服务器里的“环境”和我们的“代码”被揉在一起做成了 **镜像 (Image)**。比如我们项目里的 `Dockerfile`，就是把 JDK21 和我们的 User 服务包成了标准化集装箱。

### 扩容是怎么做的？
1. 在宿主机上，通过修改启动端口（避免相同端口冲突），再用刚才那个相同的镜像，使用不同的名字起一个新容器：
   ```bash
   docker run -d -p 8082:8080 --name leaf-service-user-2 leaf/leaf-service-user:latest
   ```
2. （如果在 Docker Compose 下）修改 `docker-compose.yml`，给微服务增加一个额外的节点宣告，再 `up -d`。

### 优势与痛点：
*   **✅ 优势**：极大地提高了发布速度！无需人工装基础软件，所有机器上的环境绝对保持 100% 一致，杜绝了“在我的电脑上没问题”的惨剧。
*   **❌ 痛点**：它依然需要你在某台具体的宿主机上去维护它！如果宿主机的硬盘撑爆了，这台机器上运行的 10 个容器还是全军覆没。你必须人工登录不同的机器集群到处敲命令，这被称为**“容器治理孤岛”**。

---

##  时代 3：Kubernetes (K8s) 云原生部署 (造物主时代)

这就是工业界的最终形态，也是我们 `infra-deploy/k8s` 目录存在的意义。K8s（Kube）不再管理“单一的容器”，它是统领千军万马（多台物理宿主机节点集群）的**云原生底层操作系统**。

### 扩容是怎么做的？(令人窒息的降维打击)

在 K8s 中，我们使用的是 **副本集缩放 (Scale)**。
不需要买机器、不需要改端口配置、不需要改网关，**只需要敲下一行指令，或者在网页上按一个按钮：**

```bash
# 将 leaf-service-user 瞬间扩容到 50 个节点应对双十一！
kubectl scale deployment leaf-service-user-deployment --replicas=50
```

K8s 收到这行命令后，**会在后台全自动执行以下魔法**：
1. **智能选址 (Scheduling)**：它会看当前集群里哪台物理宿主机（Worker Node）很闲，内存多，就把容器丢给这台机器跑。不需要人工指定跑在哪台机器上！
2. **创建与隔离 (Pod)**：利用早已在 registry 下载好的标准化镜像，像工厂印钞一样，瞬间打印出新增的 49 个隔离舱 (Pod)。
3. **服务网格动态注入 (Service/Ingress)**：不用去改 Nginx！K8s 内部的 Service 发现机制探测到新增了 49 个节点后，会自动将它们动态注册进微服务网关 (Spring Cloud Gateway / Nacos)，流量下一秒就毫无缝隙地按负载均衡砸向这 50 台机器！
4. **自愈重建 (Self-Healing)**：如果其中一台机器所在的机房网线被人拔了，K8s 会扫描并发现当前只有 49 个存活，它会立刻在另一台未断电的机器上再打印一个新的 Pod 补上，**使存活数永远等于 50！**

### 终极形态：HPA 自动弹缩
甚至连这 1 行命令都可以不用敲了！
在 K8s 里面配置一条规则（HPA）：`“如果当前 CPU 使用率超过 80%，自动加节点，最高到 100 节点；如果凌晨无人访问 CPU 掉下 20%，自动缩到 3 节点省钱。”`
系统真正具备了像呼吸一样的**弹性生命 (Elastic Compute)**！

## 📖 附录：云原生与 K8s 底层原理百问百答

针对刚接触架构的开发者，我们在此解密几个最根本、最常被问及的核心概念：

### Q1: 到底什么是“云原生 (Cloud Native)”？
“云原生”不是一种单一的技术，而是一种**构建和运行应用程序的理念**。
过去，我们写代码时总是要考虑底层操作系统是 Windows 还是 Linux，内存有多大。
进入云原生时代，它的核心主旨是：**让应用生来就是为了在云端存活的！**
* 它由几个基本支柱组成：**微服务架构** (代码解耦)、**容器化** (Docker，环境解耦)、**持续交付 CI/CD** (发布解耦)、以及**编排调度系统** (K8s，资源解耦)。
* **通俗理解**：您的应用不再关心自己运行在哪台机器的哪块硬盘上。它就像水一样，被灌入 K8s 这个巨大的池子中。池子里的水流、温度、隔离，全部由底层系统自动打理。生于云，长于云。

### Q2: 企业里的多台“宿主机”是怎么连成一个 K8s 集群的？
目前您在 Windows 本地跑的 K8s（通过 Docker Desktop 环境开启），叫 **单节点集群 (Single-Node Cluster)**。它既是老板又是打工人，所有跑应用和调度的任务都在您这一台电脑上。

**但在真实的腾讯、阿里或者银行企业内网里，它是这样部署的：**
1. **采购机器**：公司买来 100 台物理裸机（比如装了 Ubuntu 的服务器）。
2. **阵营划分**：他们会选出 3 台配置极高且极其稳定的机器，安装 K8s 的 **Master 节点软件（ApiServer, Etcd 等控制平面）**。它们是“大脑”，绝对不跑具体的业务微服务。
3. **收编小弟**：剩下 97 台机器，全部安装 K8s 的 **Worker Node 节点软件（Kubelet, Kube-proxy）**。它们是“打工人”。
4. **组网合体**：这 97 台机器启动时，会带着一个 Token 密码，向 Master 老大注册自己（命令如 `kubeadm join...`）。
5. **化零为整**：此时，在系统眼里，这 100 台原本隔离的服务器，**融合缩成了一台巨大的超级计算机**！不管一共有 1000 还是 10000 核 CPU，您只需在一台机器上敲打 `kubectl scale` 命令，Master 大脑就会通过网络指挥这 97 台打工机器中的某一台，去启动您的 `leaf-service-user` 容器。

### Q3: K8s 这么强，它是怎么抵抗高并发、怎么做负载均衡的？
当您把 `user` 服务扩容到 50 个 Pod 散布在 20 台宿主机上时，面临的最大问题是：**用户的网关或者前端，怎么知道该把请求发给哪一个具体的 Pod？它怎么知道大家到底抗不抗得住？**

K8s 的底层魔法是由几个核心组件联动实现的：

1. **Service (服务暴露与内网负载均衡)**
   当您在 K8s 中创建一个 `Service` 并把它挂在 50 个 Pod 上时，K8s 集群的每一台物理宿主机操作系统底层，都会被偷偷埋入非常多底层的防火墙规则（基于 `iptables` 或更高级的 `IPVS` 协议）。
   当一条查找 `user-service` 的流量进来时，Linux 内核不需要经过任何应用层的反向代理程序，而是直接在**操作系统网络底层**用内核态抛骰子（例如轮询算法或最小连接数），瞬间将报文丢给 50 台机器上的某一个。这种内核级的负载均衡，比依靠 Nginx 这种软件层转发要快上指数级！

2. **Ingress (七层网关分发)**
   刚才说的是集群内部通信。如果是外部公网成千上万的用户手机请求呢？
   K8s 会在集群边缘部署一个叫 Ingress Controller 的门神（底层通常集成了 Nginx / HAProxy / Envoy）。它接管了公网的唯一入口（例如双十一 App 接口域名）。门神看一眼请求头，发现是 `/api/user`，就会瞬间把请求扔给上面所说的 Service，再由操作系统底层丢给具体的 50 台机器之一。

3. **就绪探针 (Readiness Probe)**
   K8s 怎么保证请求不会派发给一个卡死的容器？
   您在配置容器时，会写一行健康检测路径。K8s 会每隔 5 秒钟不断 ping 这 50 个 Pod。一旦发现其中 1 个返回了 500 报错或者无响应，K8s 会**立刻、无情地**将它从负载均衡的列表中摘除（隔离）。外面的流量再也不会打给它，直到它恢复，或者被 K8s 杀死重建。

正是这些深埋在 Linux 操作系统内核层面的虚拟网络技术与健康摘除技术的联动，才赋予了 K8s 如此恐怖的高并发抗压实力与“怎么压都不倒”的神话！

### Q4: 既然 RocketMQ 可以按“小组 (Broker Group)”无限横向扩展，那 MySQL 和 Redis 扩容也是这样的吗？
您的架构直觉极其敏锐！**答案是：核心思想一致（都是水平切分分片），但在具体落地和操作难度上有天壤之别！**

1. **缓存：Redis Cluster 的神级扩容（哈希槽 Slot）**
   Redis 的扩容和 RocketMQ 非常像，也是**横向加组**！
   在 Redis Cluster 中，数据空间被切分成了固定的 **16384 个哈希槽 (Hash Slot)**。
   *   假设您一开始有 3 个主节点（各带 1 个从节点备用）。那么这 3 个主节点可能每人负责掌管 5000 多个槽位。
   *   **扩容时**：如果您觉得不够用了，直接再拖 1 个新的“主从节点组”加入集群。然后，敲一个 `redis-cli --cluster reshard` 迁移命令，系统就会平滑地从原来 3 个人手里总共抽调出大约 4000 个槽位，全部分发给这个新来的第 4 人！您的微服务 (Lettuce 客户端) 会自动感知这个槽位的变化，完全不需要改代码。

2. **数据库：MySQL 的地狱级扩容（分库分表 Sharding）**
   MySQL 就没那么轻松了。由于关系型数据库有强弱的事务隔离级别和复杂的表连接 (JOIN)，它**无法原生做到像 MQ 或 Redis 那样丝滑无缝的横向扩容**。
   *   如果您只是 **“读请求”** 太高撑爆了：做法比较简单，也就是我们在第 16 阶段做的 —— 维持 1 个主库 (Master)，疯狂加从库 (Slave 组)，在代码里做**读写分离**。
   *   如果您是 **“写请求”** 和 **“总数据量 (单表超千万行)”** 撑爆了硬盘：那就是整个架构界最惨烈的手术 —— **分库分表（Sharding）**。
   *   **操作方法**：您必须引入数据库中间件代理（比如 **Apache ShardingSphere** 或 **MyCat**）。然后也是去申请新机器，建一套全新的 MySQL。接着，您要在中间件里写死极具破坏性的**路由规则**，例如：`如果 user_id 是单数，请求强行路由到 1号 MySQL 集群；如果是双数，存进新买的 2号 MySQL 集群`。不仅如此，您以前写的老代码里的那些连表查询（`SELECT ... JOIN ...`）如果跨了不同的机器，会直接当场报错，必须对业务代码做重构！

**总结：**
RocketMQ 和 Redis 在设计之初，就自带了高维度的“云原生打散重组”基因；而庞大古老的 MySQL 如果走到物理级的横向切分（Sharding），无论是研发还是运维代价，都是百倍于前两者的。
