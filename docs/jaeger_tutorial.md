# 探秘微服务追踪雷达 —— Jaeger APM 大屏使用指南

当您的微服务架构从 3 个服务扩展到 30 个甚至 300 个服务时，一个普通的用户请求（如：下单）可能会穿梭几十条服务链路。如果其中某个节点报错了，或者是整个请求变慢了，您该如何排查？

靠挨个登录服务器去看日志吗？这显然是不可能的。**这时候，您就需要一个能像 X光机 一样拍下请求“全身骨骼图”的工具：这就是分布式链路追踪系统。**

在本项目中，我们使用了最新的 **Micrometer Tracing** 技术在 Spring Boot 3 应用中埋点，并将这些 X光片 数据全部送到了 **Jaeger** 大屏进行洗像与展示。

---

## 🧭 核心概念：什么是 Trace 和 Span？

在您打开 Jaeger 大屏之前，必须先掌握链路追踪世界里的 2 个绝对核心术语：

1. **Trace (调用链 / 轨迹)**：
   您可以把它理解为**“一次完整的用户请求”**。从用户点击“创建订单”按钮那一刻起，到最后页面给用户返回成功提示，这整个完整的旅程，就是一个 Trace。每一个 Trace 都会被赋予一个全球唯一的身份证号 —— **`TraceID`**。

2. **Span (跨度 / 节点)**：
   一个 Trace（大旅程）是由无数个具体的步骤（小站点）组成的，这些小步骤就是 Span。
   * 比如：请求经过网关 (Gateway)，这是一个 Span。
   * 比如：网关把请求转给订单服务 (Order)，这也是一个 Span。
   * 比如：订单服务用 OpenFeign 去拉取用户服务 (User)，这也是一个 Span。
   * 甚至：订单服务去查了一次 MySQL 数据库、写入了一次 Redis，这些通通可以被定义成一个 Span。

每一个 Span 都有自己的 **`SpanID`**，同时它还记录了：**这件事是谁做的？什么时候开始的？花了多少毫秒？执行成功没？**

**👉 一句话总结：一个 TraceID 下面挂着一串像冰糖葫芦一样的 SpanID 树。这棵树就是我们在 Jaeger 里要看的“瀑布图”。**

## 🛠️ 第一步：如何搜索您关心的全链路？

请在浏览器打开本地 Jaeger 监控台主页：**[http://localhost:16686](http://localhost:16686)**

在左侧的深色搜索栏 (Search) 中，您将使用以下武器来定位问题：

### 1. Service (服务) - 必选项
在下拉菜单中，列出了所有向 Jaeger 报道过名字的服务。
* 如果您是从网关发起的请求，通常选 `leaf-gateway` 能够抓取到最完整的全链路。
* 如果您只想看某个后端服务（如 `leaf-service-order`）自己干了些啥，也可以单独选它。

### 2. Operation (操作接口) - 选填项
选定 Service 后，这个下拉框会列出该服务对外暴露的所有 RESTful 请求路径或者内部方法（比如 `GET /api/order/create/{userId}`）。这有助于您精准过滤不想看的其他杂乱接口。

### 3. Tags (自定义标签过滤) - 杀手锏
有时候一天有十万单，您只想找出某个人报错的那一单，怎么办？
在此处输入 `error=true`，可以直接帮您把发生过报错的调用链全部过滤出来！

### 4. Lookback (时间回望) & Max Duration (最大耗时)
如果您收到报警说昨晚 8 点系统很卡，您就在这里设置过滤时间；
如果您想查“究竟是哪个调用超过了 2000 毫秒的变态耗时”，可以在 Min Duration 里填入 `2000ms`。

最后，点击最下方那颗蓝色按钮 **Find Traces (寻找调用)**。

## 📊 第二步：如何阅读时间瀑布流大盘？

点击搜索后，右侧会列出符合条件的多个 Trace（调用方块）。每一个方块的长度代表了这个请求的**总耗时**。
方块上显示 `3 Spans`，意味着这次请求经过了 3 个关键节点。
如果方块左下角有个 **⚠ 红色图标**，代表这条请求在内部发生了 Exception 报错！

**点击任意一个 Trace 方块，深入 X光图 详情页！**

### 视角：彩色甘特图 (Gantt Chart)

这是 Jaeger 最性感的灵魂所在，您将看到一排排彩色的长条形成瀑布：

1. **看宏观（长短）**：瀑布图的横轴是时间流逝。横条**越长，代表在这个节点耗时越狠**。如果您看到一条横飞出屏幕红线的长条，那您就找到了系统缓慢的绝对瓶颈（比如慢 SQL，或者网络拥堵超时的外部 API）。
2. **看树层（父子）**：您可以点开长条左边的 `>` 箭头展开子任务。比如点开 `leaf-gateway`，下方会缩进伸出 `leaf-service-order`，这代表 Gateway **调用了** Order，它们在时间轴上存在**顺延和包含**关系。
3. **看微观（详情）**：**点击任意一条彩色横条！** 侧边会弹出这个 Span 的所有秘密（`Tags` 和 `Process` 面板）：
   * `http.method`: `GET` 
   * `http.url`: 告诉您实际请求的长啥样。
   * `http.status_code`: HTTP 响应码是什么（比如 `200` 或者 `500`）。
   * `error`: 如果这里是 true，通常下方 `Logs` 面板还会直接附带把那个节点抛出的堆栈（Stack Trace）异常原原本本给您打印出来，帮您直接抓到抛错的 Java 类和行数！

## 🎯 第三步：排障组合拳实战

假设有一个小白用户跟您反馈：“老板，今天您家的网站点击下单没反应，转圈转了半天最后提示报错了！”

作为拥有一套完善 Loki + Jaeger 云原生装备的顶级架构师，您的排查操作如下：

1. **定位慢请求**：打开 Jaeger，把 Lookback 选在过去 1 小时，Min Duration 设置为 `3000ms` (3秒以上)，点搜索。
2. **锁定真凶**：屏幕上赫然出现一条长条红色的 Trace。点进去一看瀑布图，发现原来是在 `leaf-service-order` 底下的名为 `Feign: UserClient` 这个子 Span 把时间全耗光了！
3. **看破真相**：点击那根长长的 Span，在详情栏赫然看到 `error=true`，且报错信息写着 "*Timeout while reading from User Service*" (读取用户服务超时)。
4. **日志联动**：此时复制这条报错记录最上方的那串宇宙唯一 **TraceID** (比如 `63a9482f5b8a0...`)。回到我们的 **Grafana Loki 日志大屏**，在那里的搜索框直接 `"{job="microservices"} |= "那串TraceID"`，一瞬间，那台拉垮的 `User 服务器` 临死前吐出的几行最关键报错底层日志，便水落石出！

> *至此，这才是真正的工业级云原生问题排查闭环！*
